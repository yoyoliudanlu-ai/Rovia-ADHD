# 微信 Todo 监控与总结设计

## 目标

在宿主机侧增加一个常驻脚本，轮询 Supabase `todo` 表并完成三件事：

1. `todo` 到期时自动给微信 bot 发送提醒。
2. 每天生成一次中文日报并发送到微信。
3. 每周生成一次中文周报并发送到微信。

LLM 使用用户提供的 Infini GLM-5 OpenAI 兼容接口，不把密钥写进仓库。

## 为什么放在宿主机侧

这项能力需要直接访问：

- Supabase REST API
- 外部 LLM API
- 微信桥接状态

把它们放进 ESP32 固件会显著增加固件复杂度、网络耦合和排障成本。宿主机脚本可以直接复用现有 `weixin_bridge.mjs` 和 MQTT 通道，也更适合做定时轮询与本地去重状态持久化。

## 方案对比

### 方案 A：全部做进 ESP32 固件

优点：

- 单体部署

缺点：

- 固件体积和 RAM 压力明显上升
- Supabase/LLM/去重状态都不适合放在设备上
- 调试和迭代慢

### 方案 B：宿主机脚本直接调用微信 iLink API

优点：

- 不依赖 MQTT
- 路径直

缺点：

- 需要重复处理微信账号 token/收件人上下文
- 与现有桥接代码重复

### 方案 C：宿主机脚本查询 Supabase，消息仍走现有微信桥

优点：

- 复用现有 MQTT -> 微信桥接能力
- 只新增最少职责
- 更容易把主动消息与被动对话统一起来

缺点：

- 需要桥接端支持“默认联系人”或持久化最近联系人

推荐采用方案 C。

## 数据流

1. `scripts/weixin_todo_notifier.mjs` 周期性查询 Supabase `todo` 表。
2. 脚本把原始行按可配置字段映射，归一化为 `id/title/body/dueAt/isDone/updatedAt/createdAt`。
3. 到期提醒使用本地状态文件去重，避免重复发送。
4. 日报/周报在配置的本地时区时间点触发。
5. 需要总结时，脚本调用 Infini GLM-5 接口生成中文文案；失败时退回确定性模板摘要。
6. 脚本通过 MQTT 发布到 `zclaw/out`。
7. `weixin_bridge.mjs` 收到消息后发送到微信；当消息未显式指定 `user_id/ctx` 时，桥接脚本自动使用最近一次入站消息学习到的默认联系人。

## 配置策略

避免强绑定某个 `todo` schema，全部通过环境变量配置：

- Supabase URL / key / table / user UUID
- 各字段名：`id`、标题、内容、截止时间、完成字段、状态字段、创建时间、更新时间
- 日报/周报时间、时区、轮询周期
- MQTT 连接信息

这样即使实际字段名不是 `due_at`、`completed`，也不需要改代码。

## 去重与状态

本地状态文件保存：

- 已发送的到期提醒 key：`todoId + dueAt`
- 已发送的日报 key：`daily:YYYY-MM-DD`
- 已发送的周报 key：`weekly:YYYY-WW`

状态保存在用户目录下单独文件夹，避免写回 Supabase 表。

## 错误处理

- Supabase 查询失败：记录日志，下个轮询周期重试
- LLM 失败：退回本地摘要，保证日报/周报仍能发出
- MQTT 发布失败：保留待发送状态并在下次重试
- 微信默认联系人不存在：明确报错，提示先给 bot 发一条消息完成绑定
